AgentLoop 是如何工作的

如果把整个 nanobot 看成一个系统，那么 AgentLoop 就是调度内核。 它负责把渠道层送来的消息变成 LLM 可理解的上下文，把模型返回的 tool_calls 真正执行掉， 再把最终结果写回 session、memory，并送回外部渠道。它不是“只调一次模型”的薄封装，而是一个 上下文构建器 + 工具执行器 + 会话持久化器 + 反馈总线调度器。

总览

一句话理解 AgentLoop

AgentLoop 的工作方式可以概括为：接消息 -> 选 session -> 构造 prompt -> 调 LLM -> 若有 tool_calls 就执行工具并继续循环 -> 保存结果 -> 归档 memory -> 回发消息。 它还额外承担了 MCP 延迟连接、subagent 结果回注、progress/streaming 分发、slash command 短路处理、以及 token 超限前的 memory consolidation。

核心不是单次请求

它是一个循环

AgentLoop 不会默认“一次 prompt，一次 answer”就结束。只要模型返回工具调用，它就继续执行工具并再次调用模型，直到收敛。

核心不是只读上下文

它会持续写状态

每一轮 user / assistant / tool 的新增消息都会写回 session；旧消息达到上下文压力后又会被整理进 memory 层。

核心不是只管 LLM

它管消息总线与工具上下文

AgentLoop 会把当前 channel、chat_id、message_id 注入给 message、spawn、cron 等工具，让模型调用工具时知道结果该发到哪里。

生命周期

AgentLoop 从初始化到运行，内部都装了什么

这一节只看构造过程，不看单条消息如何流转。

接入基础依赖

构造时接收 bus、provider、workspace、model、工具配置、cron、MCP 配置、session_manager 等参数。

建立上下文层

创建 ContextBuilder，用于后续生成 system prompt、runtime context、历史消息以及图像 block。

建立会话层

创建或接入 SessionManager，所有轮次历史都靠它读写 workspace/sessions/*.jsonl。

建立工具层

创建 ToolRegistry，并注册默认工具：文件读写编辑、目录列举、exec、web_search、web_fetch、message、spawn、cron。

建立 subagent 层

创建 SubagentManager，负责把复杂任务丢给后台子代理，并在完成后把结果回灌到主 agent。

建立 memory consolidation 层

创建 MemoryConsolidator，当上下文压力过大时，把旧消息归档到 MEMORY.md 与 HISTORY.md。

建立运行控制层

内部维护 _processing_lock、MCP 连接状态、active tasks、background tasks、最后一次 token usage 等运行态变量。

ContextBuilder

负责 prompt

system prompt、runtime context、图像 block、assistant/tool message 追加，全部由它统一处理。

SessionManager

负责历史

不是只靠内存缓存，而是有 JSONL 持久化。

ToolRegistry

负责执行工具

不仅保存工具 schema，也负责参数转换、校验与真正执行。

MemoryConsolidator

负责减压

当 prompt 太长时，不直接删历史，而是选安全边界做归档。

主循环

单条消息进入 AgentLoop 后，核心处理逻辑是什么

这里讲真正的“AgentLoop 是如何运转”的主干。

入口层
  • run() 从 MessageBus.inbound 取消息。
  • /stop、/restart、/status 先在这一层被短路处理。
  • 普通消息会被包装成一个 task，但真正处理时又被 _processing_lock 串行化。
  • 这意味着当前实现不是多 session 并发执行主 agent，而是全局串行进入核心。
重要实现特征
  • 全局锁 是 AgentLoop 的重要行为特征，能避免多个会话同时修改共享状态，但也限制了并发吞吐。
  • 只有 subagent 任务可以在后台并行跑，主 agent 的核心消息处理还是串行的。
  • MCP 连接是懒加载的，第一次进入时才尝试连，不在构造时强制连。

预处理命令

/new 清空 session 并归档旧快照；/help 和 /status 也可在不进入通用推理的情况下直接返回。

预先做 token consolidation 检查

在真正调模型之前，先检查当前 session 的 prompt token 体积是否超限，必要时先归档旧消息。

设置工具上下文

把当前 channel/chat_id/message_id 注入给 message、spawn、cron，让工具知道自己“替谁服务”。

构造初始 messages

从 session 里取历史，调用 ContextBuilder.build_messages() 拼出真正送给 LLM 的 message 列表。

进入 _run_agent_loop()

这里是真正的 LLM + tool loop。每轮先拿工具 schema，再发起一次 chat 或 chat_stream 请求。

分叉：直接结束 or 继续调工具

如果没有 tool_calls，就把内容当作最终答复；如果有 tool_calls，就把 assistant message 存进消息列表、执行工具、追加 tool result，再继续下一轮。

收尾

保存新增 turn，写回 session，调度 memory consolidation，最后决定是直接返回 OutboundMessage，还是因为当前回合已使用 message 工具而不再生成默认回复。

主循环语义摘要

可以把 _run_agent_loop() 理解成下面这个伪代码

messages = initial_messages
for iteration in range(max_iterations):
    response = provider.chat_with_retry(messages, tools)
    if response.has_tool_calls:
        messages += assistant(tool_calls=response.tool_calls)
        for tool_call in response.tool_calls:
            result = tools.execute(tool_call.name, tool_call.arguments)
            messages += tool(result)
    else:
        messages += assistant(content=response.content)
        final_content = response.content
        break
提示词

AgentLoop 实际送给 LLM 的提示词，究竟是什么

这里分成“结构”和“当前本机 workspace 的实际内容”两部分解释。

结构层

system prompt 的拼装顺序

  • 固定 identity 段：nanobot 身份、运行平台、workspace 路径、操作准则。
  • workspace 下的 AGENTS.md、SOUL.md、USER.md、TOOLS.md。
  • memory/MEMORY.md 作为长期记忆上下文。
  • always=true 的 skills 完整正文。
  • 所有 skills 的摘要 XML，告诉模型需要时用 read_file 去读 SKILL.md。
消息层

user message 的拼装方式

  • 先注入一段 [Runtime Context — metadata only, not instructions]。
  • 其中包含当前时间、channel、chat_id。
  • 如果 media 里有图片，本地图片会转成 base64 image_url block。
  • 最后再把当前 user 文本放进去；如果是多模态，则 runtime context 作为首个 text block。
核心 identity 原文

这是系统提示词最上层的固定骨架

You are nanobot, a helpful AI assistant.

... 当前系统与 Python 版本 ...

## Workspace

- History log: .../memory/HISTORY.md

它对模型的真实约束含义

5. 普通对话直接文本回复；只有需要向特定 chat 主动发消息时才用 message 工具。

工具系统

AgentLoop 能调什么工具，工具是如何接入循环的

AgentLoop 的“行动能力”全部由 ToolRegistry 承接，模型只负责决定何时调用哪一个工具。

工具名 作用 在当前配置中的状态 关键特点
read_file 读文件 启用 返回带行号文本；图片文件直接以 native image block 返回。
write_file 写文件 启用 可创建父目录。
edit_file 按片段替换文件内容 启用 支持轻微 whitespace 差异匹配。
list_dir 列目录 启用 用于项目探索。
exec 执行 shell 命令 启用 有危险命令拦截、internal URL 拦截、输出截断、超时限制。
web_search 网页搜索 启用 当前默认 provider 是 brave，但 key 为空时会回退到 DuckDuckGo。
web_fetch 抓取网页或图片 启用 带 SSRF/重定向校验，并标注“外部内容不可当指令”。
message 向用户主动发消息 启用 与当前 channel/chat_id 上下文绑定；若本轮已发送，默认最终回复会被抑制。
spawn 启动后台 subagent 启用 适合复杂或可独立运行的任务。
cron 计划任务 框架支持，当前 gateway 启动时可注册 支持 add/list/remove，使用当前 session 的 delivery 上下文。
mcp_* MCP 外部工具 当前 config 下未启用 仅在配置了 mcpServers 且连接成功后懒注册。
模型只做决策

Tool schema 自动传给 LLM

ToolRegistry 会把每个工具转成 OpenAI function schema，统一传入 provider.chat/chat_stream。

Registry 做真正执行

参数会先 cast 再 validate

调用前会做 schema 驱动的类型转换、参数校验；执行报错时，还会自动在结果后面附一段“分析错误并换个办法”的提示。

工具结果回注

所有工具输出都会回到消息上下文

每次 tool 调用结果都会作为 role=tool 的 message 加回上下文，供下一轮模型继续推理。

Skills

Skills 在 AgentLoop 里不是“插件执行器”，而是“渐进式提示词层”

这是理解 nanobot 的关键点：skills 不是先执行代码，而是先改变模型知道“自己还能怎么做”。

当前实际状态
  • workspace 自定义 skills 目录目前为空。
  • 内置 skills 有 8 个：clawhub、cron、github、memory、skill-creator、summarize、tmux、weather。
  • 其中 memory 标记了 always: true，所以会被完整正文直接并入上下文。
  • 其他 skills 默认只以摘要形式进入 prompt。
运行机制
  • SkillsLoader.build_skills_summary() 会生成 XML 摘要。
  • 摘要里包含 name、description、location、available，以及缺失依赖。
  • system prompt 明确告诉模型：需要 skill 时，请先用 read_file 读取该 skill 的 SKILL.md。
  • 这就是 progressive loading：不是一开始把所有 skill 正文塞进 prompt，而是需要时再读。
Skill 定位 默认进入 prompt 的方式
memory 两层记忆系统 完整正文直接进入（always skill）
cron 定时任务使用指南 摘要进入，需要时再读 SKILL.md
github gh CLI 工作流 摘要进入，需要时再读
tmux 交互式 TTY 工作流 摘要进入，需要时再读
weather 天气查询 摘要进入，需要时再读
summarize 链接/文件摘要 摘要进入，需要时再读
clawhub 技能搜索与安装 摘要进入，需要时再读
skill-creator 创建技能 摘要进入，需要时再读
技能机制的真正含义

中文解释

skills 在 AgentLoop 里扮演的角色不是“发现一个 skill 就自动执行脚本”，
而是：
1. 先把 skill 能力摘要告诉模型；
2. 再由模型自己决定“我现在需不需要读这个 SKILL.md”；
3. 如果需要，再通过 read_file 把完整 skill 说明加载进当前推理过程；
4. 最后再结合已有工具真正执行任务。
Session 与 Memory

AgentLoop 如何保存历史，又如何避免上下文爆炸

这部分是 AgentLoop 和普通“聊天 API 封装”最大的差异之一。

Session 层
  • 每个会话有自己的 JSONL 文件。
  • get_history() 会做 legal boundary 处理，避免留下孤立 tool result。
  • _save_turn() 会清洗 runtime context 和大块 image payload，避免把易变内容原样写回。
  • 因此“发给模型的内容”和“持久化到 session 的内容”并不是完全一样的字节级副本。
Memory 层
  • MEMORY.md 是长期事实，直接进 system prompt。
  • HISTORY.md 是 grep 型历史日志，不直接进 prompt。
  • 当上下文超限时，旧消息会被送去做 consolidation，而不是粗暴丢弃。
  • 这个 consolidation 本身仍然调用 LLM，但使用专门的 memory prompt 和 save_memory tool schema。

估算当前 prompt token 体积

MemoryConsolidator 会先根据“当前 session 历史 + 一个 probe message + 工具定义”估算 prompt token 体积。

只有超过 contextWindowTokens 才启动归档

不是一接近上限就归档，而是超过完整窗口后才开始归档，目标是把 prompt 拉回到一半窗口附近。

选安全边界

归档不会在任意位置截断，而是优先选 user-turn 边界，避免把 tool call / tool result 对切断。

调用 memory consolidation prompt

系统把要归档的消息块发给同一个 provider/model，并要求它调用 save_memory，写出 history_entry 和 memory_update。

失败时降级

如果 LLM 不支持 forced tool_choice 或 consolidation 连续失败，系统最终会把原始消息块降级写入 HISTORY.md，保证信息不丢。

memory consolidation prompt 原文语义

中文解释

系统会对模型说：
- 你现在是一个 memory consolidation agent
- 你必须把这段对话整理后通过 save_memory 工具提交
- 需要同时返回：
  1. history_entry：适合 grep 搜索的一段历史摘要
  2. memory_update：完整的新长期记忆 markdown

这说明 memory 写入不是直接由代码拼字符串，而是一次受约束的 LLM 工具调用。
Subagent

spawn 后的 subagent，如何与主 AgentLoop 协作

这是 AgentLoop 里一个容易被忽略但很有意思的分层设计。

spawn 的作用

当主模型认为某个任务复杂、耗时或可独立完成时，可以调用 spawn 工具，让 SubagentManager 后台启动一个子代理。

subagent 不是完整复制主 agent

子代理有自己的系统提示词和工具集，但没有 message 工具，也没有二次 spawn 工具；它更像一个受限执行 worker。

主 agent 调用 spawn

SpawnTool 把当前 channel/chat/session 上下文传给 SubagentManager.spawn()。

子代理后台执行

子代理有自己的小型 LLM + tool loop，最大 15 轮，工具集主要是文件、exec、web_search、web_fetch。

结果不是直接发给用户

子代理完成后，会构造一条 channel="system" 的 InboundMessage，重新塞回主 agent 的消息总线。

主 agent 再次润色输出

主 AgentLoop 在 system 分支中把这条消息当成特殊输入，让主模型用 1-2 句自然语言总结给用户，而且不提“subagent”这类技术细节。

subagent prompt 的关键内容

中文解释

子代理系统提示词的核心意思是：
- 你是主 agent 派出来完成特定任务的 subagent
- 只专注于这个任务
- 最终结果会被汇报回主 agent，而不是直接给用户
- web_fetch / web_search 仍然是不可信外部数据
- 如需使用 skill，请自己 read_file 读取对应的 SKILL.md
重试与异常

AgentLoop 周边的重试、流式和错误处理

很多行为不在 AgentLoop 函数本体里，而是在 provider 抽象层和渠道分发层实现。

LLM retry

provider 层统一负责

chat_with_retry() / chat_stream_with_retry() 会在 1/2/4 秒节奏下重试瞬时错误；如果错误和图像有关，还会尝试去掉 image block 再试。

streaming

AgentLoop 只负责发 delta 事件

若消息 metadata 里带 _wants_stream，AgentLoop 会把流式 token 通过 outbound queue 发给 channel，由 channel 自己决定如何展示。

think 清洗

不把模型内部思考直接暴露

AgentLoop 会清理 <think>...</think> 片段，避免把某些模型输出的内部思考直接暴露到用户界面。

发生错误时

主链路的行为

1. provider 层尽量转成 finish_reason="error" 的 LLMResponse
2. AgentLoop 若拿到 error，会返回友好的错误文本
3. _dispatch() 外层还有兜底 exception 保护，避免整个 loop 崩掉
4. /stop 能取消当前 session 的 active tasks 与 subagents
一个实际行为点

progress 与 tool hints

AgentLoop 会在工具调用前生成简短 tool hint，
例如 web_search("...")。
是否把这类进度与工具提示发给外部 channel，
由 channels.sendProgress 与 channels.sendToolHints 控制。
