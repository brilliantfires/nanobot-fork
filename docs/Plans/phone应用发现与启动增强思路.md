# 后续工作思路：Android 应用发现与启动增强

## 背景

当前 `nanobot` 中的 Android 应用启动链路已经可以工作，但应用名解析仍以静态映射为主：

- `phone_launch` 先将 `app_name` 映射为包名
- 再尝试解析 launcher activity
- 然后通过 `adb shell am start` 或 `monkey` 发起启动

这条链路对于常见应用已经足够支撑当前 Phase 1 的真机验证，但它仍然有几个明显局限：

- 依赖内置应用名到包名的映射表
- 无法直接读取设备上的“可启动应用列表”
- 对应用分身、多 launcher 入口、OEM 定制包名场景支持不足
- 当应用名称不唯一时，缺少明确的候选反馈

## 当前结论

现阶段先不扩展这部分能力，优先保证：

- `adb` 连接稳定
- 设备检测稳定
- 截图、点击、返回、回桌面、启动应用等基础工具可真实执行

等到 Phase 1 的设备工具链路稳定后，再继续增强应用发现与启动机制。

## 后续增强方向

### 1. 新增设备侧应用发现能力

在 `nanobot` 中新增设备事实查询能力，而不是只依赖静态映射表。

建议新增：

- `phone apps`
  - 列出当前设备上可启动应用
  - 输出 `label`、`package`、`launcher activity`
- `phone packages`
  - 保留现有包名列表能力，作为底层排障命令

### 2. 调整应用解析顺序

后续建议将 `phone_launch` 的解析逻辑调整为：

1. 如果输入本身就是包名，则直接按包名启动
2. 如果命中静态映射，则优先尝试该包名
3. 如果未命中静态映射，则读取设备 launcher 应用列表并按 label 精确匹配
4. 如果匹配到多个候选，则明确报错并返回候选，不做静默猜测

## 3. 增强分身与 OEM 场景支持

针对应用分身和厂商定制系统，后续可考虑：

- 支持通过完整 `package/activity` 直接启动
- 支持读取多个 launcher 入口
- 对分身包名给出显式候选
- 在 CLI 中增加排障输出，帮助确认当前前台包名和启动 activity

## 4. 与当前阶段的关系

该文档记录的是后续增强方向，不属于当前 Phase 1 的必做项。

当前优先级仍然是：

- 让 `nanobot/nanobot/agent/tools/phone` 在真机上稳定工作
- 明确暴露依赖、设备、截图、动作执行失败
- 为后续 phone runner / phone subagent 打下可靠的设备层基础
