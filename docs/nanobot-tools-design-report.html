nanobot Tools 系统设计报告

1. 核心交互机制：Agent 与 Tools 的关系

在你的观察中，“Agent 会被分配一些 Tools，而 Tools 是由 Agent 来决定如何调用的” 这一理解是完全正确的。这遵循了标准的大模型 Function Calling (Tool Calling) 范式。

控制权反转： nanobot 并不会通过硬编码去依次执行工具。相反，nanobot 只是把配置好的工具清单（包含名称、描述和参数约束）“告诉”大模型（LLM）。大模型在推理过程中，如果认为需要补充额外信息或引发外部动作，就会主动返回一个 Tool Call 指令，随后 nanobot 的内核（Agent Loop / Subagent）接收指令、执行对应工具，并把结果再反馈给大模型。

2. 系统架构设计

nanobot 对 Tools 的管理是由核心基类 Tool 和注册中心 ToolRegistry 构建起来的闭环系统（位于 nanobot/agent/tools/ 包中）。

2.1 Base Tool 基类 (base.py)

所有工具必须继承自抽象方法 Tool(ABC)，强制实现以下核心属性：

  • name: 工具的唯一调用标识（比如 "spawn", "read_file"）。
  • description: 工具的文字描述。这是整个架构的“命门” —— LLM 依赖这段描述来决定是否调用该工具，以及什么场景调用。
  • parameters: 一份标准的 JSON Schema，规定了调用该工具需要传入什么参数、其数据类型及是否必填。
  • execute(**kwargs): 异步的实际逻辑执行方法。返回值可以是字符串或对象，最终会被打包送回给 Agent。

2.2 Tool Registry 注册中心 (registry.py)

这是一个动态的工具挂载器，让主 Agent 和 Subagent 可以根据职责不同“挂载”不同的工具集合：

  • 提供 register() 和 unregister()。
  • 提供 get_definitions(): 用于将当前载入的所有工具导出为 OpenAI 标准的 Schema 格式。
  • 统筹执行器 execute(name, params): 它不仅仅是转交请求，它会在执行前自动调用参数校验（验证 schema、自动转型 cast_params），并在执行失败时统一添加容错提示 [Analyze the error above and try a different approach.] 引导大模型自我修正。

3. 现有 Tools 列表

nanobot 在 nanobot/agent/tools/ 下预置了大量高原子化的原生操作工具。根据模块可以分为五大类：

📂 文件系统交互

filesystem.py: 包含 ReadFileTool, WriteFileTool, EditFileTool, ListDirTool。允许 Agent 自由探索和修改代码区或共享数据目录。

💻 命令行沙盒

shell.py: 包含 ExecTool。支持在限定的安全环境、超时机制下运行 Shell/Python 脚本操作。

🌐 网络探测

web.py: 包含 WebSearchTool, WebFetchTool。主要用于突破本地信息壁垒，实时搜索互联网并抓取正文。

💬 通信交互

message.py: 包含 MessageTool。将消息（文本或富媒体）通过底层的 MessageBus 传递给用户所在的渠道（如飞书、微信）。（注意：Subagent 通常不配备此工具以免扰民）。

🤖 并发协作

spawn.py 和 cron.py: 包含 SpawnTool (分裂子代理执行耗时任务) 和定时任务挂载。此外还包含实现 MCP(Model Context Protocol) 标准生态连接的 mcp.py。

4. 工具与 Agent 系统的融合交互生命周期

结合 agent/loop.py 和 agent/subagent.py，这里展示了一次完整的 Tool Calling 交互过程：

  1. 环境组装阶段： Agent 启动时，实例化一个 ToolRegistry()，并通过 self.tools.register(...) 方法绑定所需的工具实例。对于主 Agent 会绑定 MessageTool 和 SpawnTool；而通过 Spawn 创建的只干活的 Subagent 则只绑定基础工具，物理隔离了发图文的能力。
  2. 推演循环阶段 (Agent Loop)：
    • Agent 发起向 LLM Provider (如 Azure / OpenAI) 的请求，并在 tools 字段传入 tools.get_definitions()。
    • LLM 推理发现需要执行命令，返回含有 response.tool_calls 的数据包。
  3. 截获并执行：
    • 循环体拦截到大模型的选择，把工具名解析出来。
    • 调用 self.tools.execute(tool_call.name, tool_call.arguments)。
  4. 结果反馈： 工具执行结果被封装成一条带有 role: "tool" 和对应 tool_call_id 的 Message，压回上下文数组（Context），并在下一次 while 循环中再次发送给大模型，供其检查和分析刚刚的运行结果。这个过程最多重复 15 次（max_iterations）。

5. 附录：防御与容错设计 (数据校验机制)

为了防止大模型不可控地产出非法 JSON 或参数，Tools 系统采用了非常严格的防崩溃设计：

  • 类型强制转换 (Type Casting): 在 Tool.cast_params() 中，系统会尝试包容大模型犯下的小错误，例如将字符串类型的 "true" 按 Schema 的预期无感转换为 Bool 型的 True，将可序列化的数字变更为 Integer。
  • 边界拦截 (Validation): 执行 Tool._validate() 按 JSON Schema (必填性 required, 长度限制 minLength/maxLength, 大小范围 minimum/maximum, 枚举 enum) 进行严格查验。
  • 自我修正闭环 (Self-healing Loop): 如果数据校验没过，或者工具在执行时抛了 Exception，系统会直接拦截报错堆栈而不是崩溃。它把类似 Error: Invalid parameters for tool 'xxx'... 的信息通过 role: tool 原原本本反馈给大模型，并在末尾附加提示 [Analyze the error above and try a different approach.] 迫使 LLM 进入纠错机制。
