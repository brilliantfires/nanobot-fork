这份报告回答一个具体问题：当用户已经在 Telegram 完成与 nano-bot 的连接后， 从发送一条消息开始，这条消息在系统内部到底如何流动、会去哪里、是否会做意图分析、分析后如何处理、最终调用什么 LLM，以及提示词是如何组成的。

总览结论

最核心的答案

当前项目里，Telegram 消息 不会先经过一个单独的“意图识别服务”。它先进入 Telegram Channel，经过 ACL、群聊提及策略、媒体下载与 reply context 拼装后，被送进 MessageBus；随后由 AgentLoop 统一构造上下文并发起第一次 LLM 调用。 这一次 LLM 推理本身，就是系统真正执行“意图分析”和“动作决策”的地方。

代码直接判断的部分
  • 这是不是 Telegram 命令，如 /new、/stop、/status。
  • 群消息是否应该让 bot 响应，如是否 @bot 或回复 bot。
  • 用户是否在 allowlist 中。
  • 消息里有没有图片、语音、文档，需要不要下载与转写。
LLM 决定的部分
  • 这条自然语言消息到底是在聊天、问问题、查资料、改文件、发消息，还是需要调用工具。
  • 是直接生成最终回复，还是先触发一串 tool_calls。
  • 工具结果回来后，是否继续调用工具，还是收敛成最终答复。
一句话版本

“意图分析”存在，但不是单独模块，而是由统一 Agent 的首轮 LLM 推理完成。

实际模型版本说明

代码默认模型是 anthropic/claude-opus-4-5，但当前仓库无法单独证明你线上 Telegram bot 运行时的实际模型，因为本机默认配置文件不存在，真实实例可能由启动参数或外部 config 覆盖。

主链路

消息从 Telegram 发出后，如何一步步传递

这部分按代码执行顺序展开，展示消息从进入 Telegram bot，到离开系统回到 Telegram 的完整路径。

Gateway 启动

nanobot gateway 启动时会创建 MessageBus、AgentLoop、ChannelManager，再并发运行 agent 与各个 channel。Telegram 只是其中一个 channel。

Telegram long polling 收到更新

python-telegram-bot 通过 long polling 收消息。代码注册了命令处理器和普通消息处理器，普通消息包括文本、图片、语音、音频、文档。

Channel 层前置加工

如果是群聊，先判断是否应该响应；之后提取文本、caption、reply context，并在需要时下载媒体。语音和音频若配置了 Groq key，会先尝试转写成文本。

包装成 InboundMessage

消息被封装为 InboundMessage，携带 channel、sender_id、chat_id、content、media、metadata 和可选的 session_key_override。

进入 MessageBus.inbound

Telegram Channel 不直接调 agent，而是把消息推入异步队列。这样渠道层和 agent 核心被解耦。

AgentLoop 取出消息

AgentLoop.run() 消费 inbound queue。/stop、/restart、/status 这类命令由代码直接处理；普通消息进入统一处理流程。

加载 session 历史并组装 prompt

系统根据 telegram:<chat_id> 或 topic session key 找到会话，把历史、memory、skills、系统提示词、runtime context、当前消息和图片一起组装成 LLM 输入。

第一次 LLM 调用

这里没有独立 intent router。第一次 LLM 调用会直接判断该回复、该查资料、该调工具、还是该继续多轮工具执行。

若返回 tool_calls，就执行工具

AgentLoop 将工具调用结果写回 message 列表，再继续下一轮 LLM 调用，直到收敛为最终回复或达到最大迭代上限。

写回会话和 memory

本轮 user / assistant / tool 消息会被写入 workspace/sessions/*.jsonl；随后异步尝试把可长期保留的信息整理进 memory/MEMORY.md 与 memory/HISTORY.md。

发回 Telegram

最终结果被包装成 OutboundMessage 放入 outbound queue。ChannelManager 取出后路由到 Telegram。若打开 streaming，就先逐步编辑消息；最终再收敛到完整文本。

关键代码锚点：nanobot/cli/commands.py、nanobot/channels/telegram.py、 nanobot/channels/base.py、nanobot/bus/queue.py、 nanobot/agent/loop.py。
目标去向

这条消息会传到哪里

这里区分“必经位置”和“条件触发位置”，避免把所有外部系统都说成固定路径。

位置 是否必经 作用 备注
Telegram Bot API 是 接收用户消息，发送最终回复或流式更新 当前 Telegram channel 使用 long polling，不是 webhook。
MessageBus.inbound 是 把渠道层与 agent 核心解耦 消息先被放入 inbound queue，再由 AgentLoop 消费。
本地媒体目录 条件触发 存放下载下来的图片、语音、音频、文档 只要消息带媒体，就可能落到本地。
Groq transcription 条件触发 对语音/音频做转写 只有配置了 providers.groq.api_key 才会发生。
外部 LLM provider 普通消息基本必经 进行理解、决策、回复生成与工具调用规划 命令类如 /status、/new 可不经 LLM。
workspace/sessions/*.jsonl 普通消息处理后会写入 持久化会话历史 会存 user / assistant / tool 记录。
memory/MEMORY.md 与 HISTORY.md 异步条件触发 长期记忆与历史归档 由 memory consolidator 根据 token 阈值与归档逻辑执行。
Web / Exec / MCP / Cron 等工具目标 条件触发 只有模型决定调用对应工具时才会访问 这是“分析后的动作执行面”，不是所有消息都会走到这里。
一个重要细节

图片和非图片媒体进入 LLM 的方式不同

图片会在 prompt 中被转成 base64 image_url block，作为真正的多模态输入；但语音、音频、文档、视频不会被原样送给 LLM，它们通常以转写文本或路径标签的形式进入内容。

reply 上下文也会混入当前消息

回复某条 Telegram 消息时

系统会把被回复消息的文本或媒体摘要前置成形如 [Reply to: ...] 的内容，因此 LLM 看到的“当前 user message”实际上已经不是 Telegram 原文本，而是处理后的组合消息。

意图分析

会不会做意图分析，分析后怎么处理

这里把“硬编码规则判断”和“LLM 驱动的意图决策”拆开说明。

规则判断

第一层：代码直接分流

如果消息是 /new、/status 等 slash command，代码直接返回对应结果，不走通用自然语言理解。群聊里是否理会，也由 Telegram channel 先判断。

语义分析

第二层：LLM 进行意图决策

普通文本不会先进入一个单独的 intent classifier。系统是把完整上下文直接送给 LLM，由模型判断需要直接回答，还是需要读文件、执行命令、搜索网页、发送消息、调用 MCP 等。

执行循环

第三层：按 tool call 驱动后续动作

如果模型返回 tool_calls，系统就执行工具、把结果追加回上下文，再继续调 LLM，直到获得最终回复。这就是“分析之后怎么处理”的核心机制。

系统里不存在的东西
  • 没有单独名为 IntentService 或 Classifier 的模块。
  • 没有单独保存 intent=xxx 这样的结构化意图字段。
  • 没有在 Telegram 层先做 LLM 分类、再决定是否进入主 agent。
系统里实际存在的东西
  • slash command 的显式代码处理。
  • 群消息是否响应的显式代码处理。
  • 统一的 AgentLoop + LLM + tool execution 循环。
  • 模型根据上下文输出“最终回复”或“工具调用”的能力。
行为结果

分析之后有两条出口

出口 A：模型直接返回普通 assistant content
-> AgentLoop 结束本轮
-> 保存 session
-> 发布 OutboundMessage
-> Telegram send()/send_delta()

出口 B：模型返回 tool_calls
-> ToolRegistry.execute(...)
-> tool result 追加回 messages
-> 再次调用 LLM
-> 重复直到得到最终 assistant content 或超出 max iterations

if provider_name == "openai_codex" or model.startswith("openai-codex/"):
    OpenAICodexProvider(...)
elif provider_name == "custom":
    CustomProvider(...)

提示词

使用什么提示词

这里的 prompt 不是单个硬编码字符串，而是动态拼装的“系统提示词 + 历史 + 运行时上下文 + 当前用户消息”。

System Prompt 的组成
  • 固定身份段： nanobot 身份、运行平台、workspace 路径、操作约束。
  • workspace bootstrap files： AGENTS.md、SOUL.md、USER.md、TOOLS.md。
  • 长期记忆： memory/MEMORY.md。
  • 技能信息： always-on skills 内容 + skills summary。
User Message 的组成
  • runtime context： 当前时间、channel、chat_id。
  • 当前消息正文： 文本、caption、reply context 拼接后的最终字符串。
  • 图片媒体： 若本地文件是图片，会编码为 base64 image_url block。
  • 语音/文档等非图片媒体： 不会直接当成多模态块送入模型，通常只留下转写文本或文件路径标签。

返回路径

分析完成后，结果如何回到 Telegram

用户看到的回复，不是 AgentLoop 直接发给 Telegram，而是再经过一层 outbound bus 与 channel manager。

1. 生成 OutboundMessage

AgentLoop 结束本轮

如果本轮没有通过 message 工具主动向当前会话发消息，AgentLoop 会构造一个 OutboundMessage，携带 channel、chat_id、content 与 metadata。

2. 进入 MessageBus.outbound

统一交给渠道管理器

ChannelManager._dispatch_outbound() 从 outbound queue 取消息，并根据 msg.channel 找到对应 channel 实例。

3. Telegram send 或 send_delta

回到最终聊天窗口

如果 metadata 表示流式更新，就走 send_delta() 持续编辑消息；否则走 send() 一次性发送完整文本。

streaming

Telegram 默认支持渐进式回复

Telegram config 里默认 streaming = true。如果启用流式输出，用户在 Telegram 上通常会看到一条逐步被编辑完善的消息，而不是等全部完成后一次性返回。

reply_to 行为

是否回复到原消息取决于配置

只有当 Telegram channel 的 reply_to_message 为真时，bot 发回的消息才会带上 reply 参数，指向原始 Telegram message_id。

限制与不确定项

哪些结论是确定的，哪些结论需要额外配置才能确认

为了避免把推断说成事实，这里明确列出当前研究的边界。

可以确定的部分
  • Telegram 使用 long polling 接入，而不是 webhook。
  • 消息会经由 MessageBus、AgentLoop、ChannelManager 组成的统一总线处理。
  • 不存在独立的 intent classification service。
  • LLM 与工具调用循环是自然语言处理的核心机制。
  • session 会落盘到 workspace 的 sessions 目录。
  • memory consolidation 会写入 memory/MEMORY.md 与 memory/HISTORY.md。
当前无法仅凭仓库确认的部分
  • 你线上 Telegram bot 当前实际使用的是哪个 model/provider。
  • 你的实际 workspace 中 AGENTS.md、USER.md 等是否被改写过。
  • Telegram channel 的真实 allowlist、group policy、reply_to_message、streaming 开关值。
  • 是否配置了 Groq API key，从而让语音自动转写。
  • 是否配置了 MCP servers、web search provider 等外部能力。
最关键的不确定项

“调用什么 LLM”不能只看代码默认值。 当前代码默认模型是 anthropic/claude-opus-4-5，但真实实例往往由 config 覆盖。 这次检查里，默认路径 ~/.nanobot/config.json 不存在，所以报告必须把“默认值”和“实际值”分开陈述。
